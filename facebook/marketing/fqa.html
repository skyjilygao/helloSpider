固定連結
審查團隊「無法重現權限」，這是什麼意思？

審查程序期間，我們的審查團隊會遵循您的指示，以重現權限在您的應用程式內的使用方式。如果我們無法重現此體驗，例如，因為我們無法遵循您的指示或者無法登入您的應用程式，那麼我們也無法批准提交。

若要避免，請：

  • 提供使用該權限的可運作的應用程式版本
  • 確定您在「新增備註」區段提供明確的指示
  • 確定所要求的登入權限可個人化使用者體驗而且遵循我們的原則

特別是對於 publish_actions 權限，請確認您的應用程式發佈功能設定正確。在審查程序期間，我們需要能夠將您的應用程式內容發佈回 Facebook。

固定連結
我的應用程式被拒絕的原因是「一般問題」， 這是什麼意思？

應用程式審查程序涉及在每個支援的平台上載入您的應用程式、使用 Facebook 登入，以及在審查中使用您要求的每個 Facebook 整合。這經常會導致我們所說的「一般問題」。這些錯誤和載入您的應用程式、登入您的應用程式，或者您的應用程式的一般性功能相關。這表示我們無法測試您在提交中要求的權限。

由於這些問題會導致我們無法審查您的 Facebook 功能，我們無法詳細評論您的應用程式如何使用您提交審查的 Facebook 功能。因此，我們以「一般問題」拒絕，並就每個平台對此提供意見。

如果您收到「一般問題」拒絕，請仔細閱讀所有意見。每個平台都會收到個別意見，說明審查時遇到的問題。

如果我的某個權限未獲得批准，會出現什麼情況？

提供您的審查回應將包含您的應用程式未獲得批准的明確說明，以及您接下來應採取的步驟。我們希望您盡快通過此程序，因此，請仔細閱讀此意見。您完成必要變更後，即可以重新提交審查。

如果您的應用程式以不會獲得批准的方式使用權限，提供給您的意見中會包含相關說明，且您不應重新提交審查。

若要通過應用程式中心的相關批准，您的應用程式必須符合我們的資格要求。符合 Facebook 應用程式中心資格的應用程式必須使用「Facebook 登入」或者具有 Facebook 畫布應用程式。

符合資格可在應用程式中心中列出的應用程式如下：

  • 使用 iOS 專用 Facebook 登入或 Android 專用 Facebook 登入建置的 iOS 或 Android 流動應用程式
  • 使用 Facebook 登入的網絡或流動應用程式

您的文字資源和宣傳形像也必須符合 我們的準則。

我需要 user_likes 以瞭解用戶是否在我的頁面上讚好。 為什麼無法獲得批准？

它違反平台政策第 4.5 節鼓勵用戶使用社交附加程式或者在專頁讚好的規定。其中包括根據用戶是否在專頁讚好來提供獎勵或對應用程式或應用程式內容設限。User_likes 不會被批准用於這種目的。

為了確保優質關係鏈以及協助公司發掘重要用戶，我們希望用戶是因為想進一步聯繫或者取得公司資訊才讚好專頁，而不是為了人為的獎勵。我們相信這個原則能夠讓用戶以及廣告客戶等獲益。

您移除任何預先填入元素並只分享用戶自行撰寫的內容之後，您應該重新提交審查，同時附上顯示內容如何從您的應用程式分享至 Facebook 的螢幕錄影。

我提供的憑證可以運作，但審查員卻無法登入。我應該怎樣做？

我們的審查團隊可能需要其他程式登入憑證才能夠登入您的應用程式，以完成審查。

如果您的應用程式在「Facebook 登入」之前或之後需要輔助登入，請確定提供該登入所需的用戶名稱和密碼。這可能包括測試或示範伺服器所需的登入憑證，以及您的應用程式或電郵註冊流程所需的輔助登入。

位於預備或開發伺服器上的應用程式可能需要進一步登入，才能夠存取您的伺服器。請同時針對此項作業提供所有必要的登入憑證。

如果您仍然不確定遺漏哪些憑證，您可以提供影片，內含您接下來提交時應該包含的「Facebook 登入」選項和所有相關的 Facebook 整合。

固定連結
應用程式在我這裡正常運作，但在審查員那裡卻不是如此。 接下來該怎樣做？

我們的審查團隊需要登入您的應用程式並檢查所有 Facebook 整合，才能夠批准您提交的應用程式。

如果審查員無法載入或使用您的應用程式，請確定：

  • 應用程式網址可被公開存取而且未設定為 localhost
  • 您已經提供存取您的開發或預備網站所需的用戶名稱和密碼
  • 您的網站具有最新的安全憑證，而且對新用戶不會造成錯誤。
  • 您能夠以新建的測試用戶身分登入和使用您的應用程式
  • 您提交審查的項目屬於附加項目而且正在您的應用程式中運作

如果您因相同原因再度被拒絕，請更新審查指示或新增備註區段，要求審查員提供更明確的說明以及其他資訊。

如果您認為審查員拒絕您的應用程式並不正確，您應該更新審查指示，在其中提供更多資訊給審查員，然後重新提交審查。

否。權限一旦獲得批准，您即可在任何平台的任何應用程式版本上使用該權限。

如果您是在新平台上拓展和開發應用程式，將不需要提交審查。您只有在要求新權限時才需要重新提交審查，例如在應用程式加入新功能時。變更和提交「應用程式詳細資料」或「開放式圖表」動作不會影響您已獲得批准的權限。

用戶要如何邀請朋友使用我的應用程式？

如果您的應用程式是遊戲並在 Facebook 畫布呈現

您可以使用下列功能邀請新玩家加入您的遊戲：

  • 要求對話框。使用要求對話框時，您可以設定「filters=app_non_users」來篩選對話框，使其僅顯示未使用您應用程式的用戶。如果您的應用程式具備畫布功能，您還可以在 iOS 和 Android 上使用「要求」對話框。
  • 可邀請的朋友 API。如果您的應用程式是一款遊戲而您想要自行建立多位朋友選擇器，您可以使用可邀請的朋友 API，這會傳回用戶朋友中未使用應用程式的排名清單。當用戶選擇要邀請的朋友時，您可以將可邀請的朋友 API 傳回的憑證傳送至要求對話框的欄位，供用戶從中傳送邀請給朋友。

如果您的應用程式未在 Facebook 畫布呈現

您可以使用 iOS 訊息對話框和 Android 訊息對話框或網頁版發送對話框。這些產品可讓用戶將內含您的應用程式連結的訊息直接發送給朋友。

這種類型的訊息可直接和少數用戶溝通，是絕佳的溝通渠道。訊息對話框和發送對話框都包含輸入提示，可讓用戶方便選擇多位要接收邀請的朋友。

為什麼 Facebook 要在伺服器端驗證流程中的 redirect_uri 附加「#_=_」？

因為可以將驗證流程重新導向至另一個流程，所以也可以利用其他的應用程式存取某個應用程式的敏感性驗證資料。透過在重新導向網址中附加新的井號片段的方式，就可以防止發生這種瀏覽器行為模式。如果因為美觀理由或用戶端行為而不喜歡這樣的網址，也可以使用 window.location.hash（或甚至您自己的伺服器端重新導向網址）來移除不雅的字元。

從我的 Facebook 應用程式分享圖像至 Facebook 時，是否也能分享自訂文字？

分享內容至 Facebook 時，預先填入文字區塊有違 Facebook 政策。您的應用程式用戶必須自行填入他們要分享的文字。

在分享內容時，預先填入文字區塊違反了平台政策 2.3 的規定 ( https://developers.facebook.com/policy/#control )。我們制定這個政策的目的是為了確保用戶在 Facebook 上分享的內容完全是他們自己的想法，避免不小心出現未經用戶同意的文字內容。
